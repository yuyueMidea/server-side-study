# 一、系统总体定位（先统一认知）

👉 本系统本质是：

> **以“单据驱动”的库存 + 资金流系统**

核心原则：

* 一切业务 = 单据（订单 / 入库 / 出库 / 退货）
* 一切库存变化 = **由单据驱动**
* 一切财务变化 = **由业务单据映射**

---

# 二、总体架构设计（推荐分层 + 可演进微服务）

## 2.1 架构选型

### ✅ 推荐：**模块化单体（Modular Monolith）起步**

原因（非常关键）：

* PSI 强事务一致性（库存 + 资金）
* 微服务容易“超卖 + 数据不一致”
* 早期业务不复杂 → 单体更高效

👉 后期可演进：

```
单体 → 模块拆分 → 微服务
```

# 四、核心设计思想（必须理解）

## 4.1 单据驱动模型（最关键）

系统所有行为都围绕：

| 类型  | 示例        |
| --- | --------- |
| 入库单 | 采购入库、退货入库 |
| 出库单 | 销售出库、调拨出库 |
| 订单  | 采购订单、销售订单 |

👉 库存 = **所有单据累计结果**

---

## 4.2 状态机（每个单据必须有）

例如：采购订单

```
草稿 → 已提交 → 已审核 → 部分入库 → 完成 → 关闭
```

👉 强制：

* 状态不可逆（防数据错乱）
* 所有操作必须校验状态

---

## 4.3 库存模型（核心设计）

库存必须拆成：

```
实际库存 = 可用库存 + 锁定库存
```

| 类型   | 含义     |
| ---- | ------ |
| 可用库存 | 可销售    |
| 锁定库存 | 已下单未出库 |

---

## 4.4 财务不是独立系统（而是映射）

👉 关键设计：

```
业务单据 → 财务流水
```

例如：

| 业务   | 财务 |
| ---- | -- |
| 销售出库 | 应收 |
| 收款   | 收入 |
| 采购入库 | 应付 |

## 5.1 基础数据模块（Master Data）

### 核心实体

* Product（商品）
* SKU（规格）
* Supplier（供应商）
* Customer（客户）

```
SPU + SKU
```

👉 价格体系：

* 标准价
* 客户等级价
* 最近采购价

---

## 5.2 采购模块（Purchase）

### 核心流程

```
采购订单 → 入库 → 付款
```

### 核心单据

* PurchaseOrder
* PurchaseInbound
* PurchaseReturn

### 关键点

* 支持部分入库
* 支持多次收货
* 自动生成应付账款

---

## 5.3 销售模块（Sales）

### 核心流程

```
报价 → 销售订单 → 出库 → 收款
```

### 核心单据

* SalesOrder
* SalesOutbound
* SalesReturn

### 关键点

* 下单锁库存
* 出库扣库存
* 自动生成应收

---

## 5.4 库存模块（Inventory）

### 核心功能

* 库存台账（必须有）
* 调拨
* 盘点

### 核心表

```
inventory_stock       当前库存
inventory_log         变动记录（最重要）
```

👉 所有库存变更必须写 log（审计用）

---

## 5.5 财务模块（Finance）

### 核心模块

* 应收（AR）
* 应付（AP）
* 资金流水（CashFlow）

### 核心设计

```
不直接修改余额 → 只记录流水
```

👉 余额 = 流水汇总

---

## 5.6 报表模块（Reports）

### 分两类：

#### 1️⃣ 实时报表（查库）

* 当前库存
* 今日销售

#### 2️⃣ 离线报表（推荐）

* 使用定时任务 + 统计表

👉 技术建议：

* MySQL + ClickHouse（后期）

# 七、关键技术设计（Go 特性结合）

## 7.1 并发控制（库存核心）

👉 必须解决：

### ❗ 超卖问题

解决方案：

* 数据库乐观锁（version）
* Redis 分布式锁（高并发）

---

## 7.2 事务设计

👉 推荐：

```
强事务：单库 MySQL
```

避免：

* 分布式事务（早期不要）

---

## 7.3 异步架构（可选）

使用：

* Kafka / RabbitMQ

用于：

* 报表统计
* 通知
* 日志

---

## 7.4 缓存设计

Redis：

* 商品缓存
* 库存缓存（慎用，需一致性策略）

# 十一、系统难点总结（你开发时会踩的坑）

我直接帮你标出来：

### 1️⃣ 库存一致性（最大坑）

* 超卖
* 并发扣减

---

### 2️⃣ 单据状态混乱

👉 必须严格状态机

---

### 3️⃣ 财务对不上

👉 一定：

```
业务驱动财务，不允许手动改余额
```

---

### 4️⃣ 报表性能

👉 不能直接查业务表
